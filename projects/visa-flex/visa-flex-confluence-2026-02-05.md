##  Overview

**Visa Flex** (also referred to as **Zilch Flex** internally) is a strategic project leveraging **Visa's Flexible Credentials (VFC)** product to enable Zilch to dynamically route transactions between consumer and commercial BINs. This allows Zilch to optimise interchange revenue while simplifying the customer experience.

---

##  Business Objective & Hypothesis

> **If we** integrate with Visa's Flexible Credential solution, **we will** increase in-store interchange revenue, **because** we can seamlessly route transactions between consumer and commercial BINs, earning commercial interchange rates on eligible in-store transactions from affiliate merchants.

---

##  Current Problems Being Solved

| Problem | Description |
|---------|-------------|
| **Lower Interchange on In-Store Transactions** | Consumer BIN transactions (physical card/mobile wallets) generate ~30bps vs. commercial BIN transactions (online) generating up to ~185bps |
| **Intra-EEA Volume Risk** | Visa plans to restrict commercial BINs for non-affiliated EEA merchants, risking declined transactions |
| **Complex Card Infrastructure** | Customers currently have different credentials for online (commercial) vs. in-store (consumer), preventing a "one-card experience" |

---

##  Projected Financial Impact

| Metric | Current | With Visa Flex |
|--------|---------|----------------|
| **Blended Interchange Rate** | 93bps | 110bps |
| **Monthly GMV Uplift** | - | **300K** |

##  Proposed Solution

### High-Level Architecture

Integrate with **Visa Flex "Self-Serve"** model, where:
- All **consumer cards** become **primary credentials** (exposed to customers)
- All primary credentials link to a **secondary commercial credential** (hidden from customers)
- Zilch dynamically routes **"in-network"** (affiliate merchant) transactions to the commercial credential during authorisation

### BIN Configuration

| BIN Type | BIN Number | Status | Flex Role | Use Case |
|----------|------------|--------|-----------|----------|
| **Consumer Credit** | 44135387 | Live | Primary | Mobile wallets, in-app, physical card, tokenisation |
| **Commercial Corporate Credit (C)** | 44136916 | New | Secondary | Hidden credential for "in-network" affiliate transactions |
| **Commercial Purchasing Credit (P)** | 44135388 | Existing | Not Flex-compatible | Maintained for legacy COF transactions only |

### Card Issuing Model

**For New Customers:**
- Primary  Virtual Consumer Card (in-app, wallet provisioning)
- [Optional] Primary  Physical Consumer Card
- Secondary  Commercial (C) Card (used for flexing, never exposed)

**For Existing Customers:**
- Consumer cards become primary credentials
- New Commercial (C) cards issued to all existing customers
- Commercial (P) cards retained for existing Card-on-File (COF) transactions

##  Key Technical Requirements

### Payment Processing

| Scenario | Behaviour |
|----------|-----------|
| **In-Network Authorisation** | If Visa flags as "Flex eligible" + merchant is affiliate  approve with Commercial (C) credential |
| **Out-of-Network Authorisation** | Approve with primary Consumer credential |
| **Refunds/Reversals** | Must return the funding source used in the original transaction |
| **3DS Authentication** | Always performed on primary credential (doesn't affect authorisation) |

### Merchant-Initiated Transactions (MITs)

**Flex Eligible MITs:**
- Delayed charges, Instalments, No-show, Recurring, Unscheduled COF

**Non-Flex Eligible MITs (must use original credential):**
- Incremental authorisations, Pre-authorisations, Resubmissions

### Product Eligibility Inquiry (PEI)
- Mandatory for Self-Serve Issuers
- 0100 Authorization message with processing code 39
- Secondary credential in PEI response must match authorisation if:
  - Amount is within 5% of anticipated
  - Time elapsed is within 30 seconds

##  Key Takeaways

1. **Revenue Opportunity**: Visa Flex unlocks ~300K/month additional interchange from in-store affiliate transactions
2. **Customer Experience**: Simplifies to a single consumer credential for all transaction types
3. **Self-Serve Model**: Zilch retains control over routing decisions based on affiliate merchant status
4. **No Acquirer Changes Required**: VFC is transparent to merchants and acquirers
5. **Target Launch**: May 2026, with certification testing in March-April 2026
